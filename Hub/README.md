# TypeScript Design Patterns — Hub

An interactive browser-based hub for all 21 TypeScript design pattern demos in this repo. Browse patterns by category, search by name or description, and open any demo directly in an embedded iframe alongside its full documentation.

---

## Running Locally

The Hub runs on port **9001** so it can coexist with any individual pattern demo (which all use port 9000).

**Prerequisites:** Build each pattern before starting the Hub so the iframes have something to load.

```bash
# 1. Build all patterns from the repo root
for dir in Behavioral/*/ Creational/*/ Structural/*/; do
  [ -f "$dir/package.json" ] && (cd "$dir" && npm install && npm run build)
done

# 2. Start the Hub
cd Hub
npm install
npm start          # opens http://localhost:9001
```

To run a single pattern on its own dev server (hot-reload):

```bash
cd Behavioral/ChainOfResponsibility
npm start          # opens http://localhost:9000
```

## Project Structure

```
Hub/
  src/
    index.ts         # Rendering logic, tab switching, iframe loading
    patterns.ts      # All 21 patterns — metadata + full markdown content
    template.html    # Webpack HTML template (separate from output index.html)
    css.d.ts         # CSS module type declaration
  style.css          # Dark theme — cards, modal, markdown styles
  index.html         # Built output (generated by webpack — do not edit directly)
  webpack.config.js  # Port 9001; also serves repo root so iframe paths resolve
  package.json
  tsconfig.json
```

### Key architectural notes

- **Iframe paths** use `./` relative URLs (e.g. `./Behavioral/ChainOfResponsibility/index.html`) so they resolve correctly both on localhost and on GitHub Pages, where the site lives at a subpath.
- **Separate template file** (`src/template.html`) prevents `HtmlWebpackPlugin` from accumulating duplicate `<script>` tags across multiple builds — a known issue when the template and output are the same file.
- **`marked`** (npm) renders the markdown documentation bundled in `patterns.ts`.

## Deploying to GitHub Pages

A GitHub Actions workflow is already included at `.github/workflows/deploy.yml`. It builds every pattern and the Hub, assembles the output, and deploys to Pages on every push to `main`.

### Step 1 — Push to GitHub

If you haven't already:

```bash
git remote add origin https://github.com/<user>/<repo>.git
git push -u origin main
```

### Step 2 — Enable GitHub Pages

In your GitHub repository:

> **Settings → Pages → Build and deployment → Source → GitHub Actions**

Save. That's it — no branch or folder to configure.

### Step 3 — Trigger the workflow

The workflow runs automatically on every push to `main`. To trigger manually:

> **Actions → Deploy Hub to GitHub Pages → Run workflow → Run workflow**

The build takes roughly 3–5 minutes (20 pattern builds + Hub build).

### Step 4 — Visit your live site

```
https://<user>.github.io/<repo>/
```

The Hub loads with all pattern demos available in iframes — no server required.

## How the Deployment Works

The workflow (`.github/workflows/deploy.yml`) does the following:

1. **Builds all patterns** — runs `npm ci && npm run build` in each pattern directory that has a `package.json`. The Adapter pattern is skipped (no webpack setup).
2. **Builds the Hub** — runs `npm ci && npm run build` in `Hub/`.
3. **Assembles `_site/`** — copies the Hub's `index.html` + `dist/index.js` to the root of `_site/`, then copies each pattern's `index.html`, `style.css`, and `dist/index.js` into matching subdirectories:

   ```
   _site/
     index.html                                   ← Hub
     dist/index.js                                ← Hub bundle
     Behavioral/
       ChainOfResponsibility/
         index.html
         style.css
         dist/index.js
       Command/ ...
       (8 more)
     Creational/
       AbstractFactory/ ...
       (3 more)
     Structural/
       Bridge/ ...
       (6 more)
   ```

4. **Deploys** — uploads `_site/` as a Pages artifact and deploys it using the official `actions/upload-pages-artifact` + `actions/deploy-pages` actions.

The Hub's `node_modules` are cached between workflow runs (keyed on `Hub/package-lock.json`). Pattern `node_modules` are not cached since each pattern has its own lock file.

## Troubleshooting

| Symptom | Likely cause | Fix |
|---|---|---|
| Demo iframe is blank | Pattern wasn't built before starting Hub | Run `npm run build` in the pattern's directory |
| Demo iframe is blank on GitHub Pages | Workflow build failed for that pattern | Check the Actions log for build errors |
| 404 on the GitHub Pages URL | Pages source not set to GitHub Actions | Settings → Pages → Source → GitHub Actions |
| Workflow doesn't run on push | Pushed to a branch other than `main` | Merge to `main`, or edit the workflow `branches` list |
| Duplicate `<script>` tags in a pattern | Pattern was built multiple times with template = output | Create `src/template.html` and update `webpack.config.js` to use it (already fixed for affected patterns) |

## Adding a New Pattern to the Hub

1. Build the new pattern (`npm run build` in its directory).
2. Add an entry to `Hub/src/patterns.ts` following the existing `Pattern` object shape.
3. Rebuild the Hub (`npm run build` in `Hub/`).
4. The new card appears automatically — no changes needed to the HTML or CSS.
